Case Study · 05

SQL Analytics

01

Problem

The admissions operation ran across multiple systems with no single query layer. Basic insights — counsellor performance, publisher ROI, funnel drop-off — required manual Excel work and were not available in real time.

02

Approach

Wrote production SQL queries using LEFT JOINs, CASE WHEN aggregations, NULLIF for safe division, and SUBSTRING_INDEX to clean messy lead-owner name strings — joining leads, applications, and enrolment tables into unified insight layers.

03

Sample Queries

counsellor_scorecard.sql 
SELECT
    TRIM(SUBSTRING_INDEX(leads.lead_owner, '(', 1)) AS lead_owner,
    -- Lead volume
    COUNT(*) AS total_leads,
    COUNT(CASE WHEN leads.lead_status = 'VERIFIED LEADS'   THEN 1 END) AS vl_leads,
    COUNT(CASE WHEN leads.lead_status = 'UNVERIFIED LEADS' THEN 1 END) AS ul_leads,
    -- Application funnel
    COUNT(applications.registered_name)                                          AS total_apps,
    COUNT(CASE WHEN applications.payment_status = 'PAYMENT PENDING'  THEN 1 END) AS total_npa,
    COUNT(CASE WHEN applications.payment_status = 'PAYMENT APPROVED' THEN 1 END) AS total_pa,
    COUNT(CASE WHEN applications.enrolment_status = 'YES'             THEN 1 END) AS total_adms,
    -- Conversion rate (leads → paid applications)
    ROUND(
        COUNT(CASE WHEN applications.payment_status = 'PAYMENT APPROVED' THEN 1 END) * 100.0
        / NULLIF(COUNT(*), 0),
    2) AS pa_conversion_percent
FROM leads
LEFT JOIN applications
    ON leads.lead_id = applications.lead_id
GROUP BY
    TRIM(SUBSTRING_INDEX(leads.lead_owner, '(', 1))
ORDER BY
    total_pa DESC;

Cleans messy lead-owner strings like "Ravi Kumar (BLR)" → "Ravi Kumar" using TRIM + SUBSTRING_INDEX

Tracks each counsellor's full funnel — total leads, verified leads, apps, pending, paid, and enrollments

Calculates lead-to-paid conversion % with NULLIF to safely handle zero-lead counsellors

Joins on lead_id and ranks output by paid applications to surface top performers

publisher_funnel.sql 
SELECT
    leads.publisher_name,
    -- Lead volume
    COUNT(*) AS total_leads,
    COUNT(CASE WHEN leads.lead_status = 'VERIFIED LEADS'   THEN 1 END) AS vl_leads,
    COUNT(CASE WHEN leads.lead_status = 'UNVERIFIED LEADS' THEN 1 END) AS unvl_leads,
    -- Application funnel
    COUNT(applications.registered_name)                                              AS total_apps,
    COUNT(CASE WHEN applications.payment_status = 'PAYMENT PENDING'  THEN 1 END) AS pending_apps,
    COUNT(CASE WHEN applications.payment_status = 'PAYMENT APPROVED' THEN 1 END) AS paid_apps,
    COUNT(CASE WHEN applications.enrolment_status = 'YES'             THEN 1 END) AS adms,
    -- Conversion rates
    ROUND(
        COUNT(CASE WHEN applications.payment_status = 'PAYMENT APPROVED' THEN 1 END) * 100.0
        / NULLIF(COUNT(applications.registered_name), 0),
    1) AS app_to_paid_pct,
    ROUND(
        COUNT(CASE WHEN leads.lead_status = 'VERIFIED LEADS' THEN 1 END) * 100.0
        / NULLIF(COUNT(leads.registered_name), 0),
    1) AS lead_verify_pct,
    ROUND(
        COUNT(CASE WHEN applications.enrolment_status = 'YES' THEN 1 END) * 100.0
        / NULLIF(COUNT(CASE WHEN applications.payment_status = 'PAYMENT APPROVED' THEN 1 END), 0),
    1) AS paid_to_enrolled_pct
FROM leads
LEFT JOIN applications
    ON leads.user_id = applications.user_id
GROUP BY leads.publisher_name
ORDER BY paid_apps DESC;

Groups the entire lead-to-enrollment funnel by publisher — showing which channels actually convert vs. just generate volume

Calculates 3 independent conversion rates: app→paid, lead→verified, and paid→enrolled per publisher

Uses NULLIF on all denominators to prevent division-by-zero errors in production

Joins on user_id (vs. lead_id in counsellor query) — exposing cross-table relationships
